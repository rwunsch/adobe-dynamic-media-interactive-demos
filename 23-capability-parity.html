Dynamic Media OpenAPI · Classic vs OpenAPI

Same intent, two delivery tiers

The same master asset, delivered by Classic Scene7 (s7g10.scene7.com/is/image/RobertCo/forest-trail) and by Dynamic Media OpenAPI (delivery-…adobeaemcloud.com/…/as/forest-trail.jpg). Pick a transform — it renders on both tiers, live, with both URLs. The OpenAPI tier deliberately exposes a lean, governed subset of modifiers; the rich Scene7 vocabulary stays on Classic. Everything here is probed live against the real tenant.

The two parameter dialects

Classic uses wid / qlt; OpenAPI uses width / quality. crop=x,y,w,h and rotate share the same syntax on both. Format is an extension (.webp) or Accept-negotiated on OpenAPI; a Smart-Imaging ruleset on Classic.

Still Classic-only

The rich Scene7 modifier vocabulary isn't (yet) on the OpenAPI delivery tier. Where it's live-checkable, the OpenAPI response is probed in real time — e.g. a named smart crop returns a real 400 Invalid Parameters.

Why a lean subset? The OpenAPI tier is built for governed, headless, multi-channel delivery — a stable ID, approval-gated, with just the transforms a web/app client needs. Heavy art-direction (named smart crops, image presets, colour treatments, text/layer compositing, interactive viewers, Basic Templates, 3D) lives in Classic. Adobe is steadily closing the gap — video Smart Crop and multi-caption/audio reached OpenAPI in 2026.5.0.
